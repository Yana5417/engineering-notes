# 前端工程师如何用 OpenSpec + Claude 两周交付一个 Go 后端服务

> 我是一个写了几年 React 的前端开发。两周前，我接到任务：基于架构师提供的设计文档，独立完成公司计量计费服务（Go + Gin + MySQL）的全部接口开发和 K8S 部署。两周后，68 个 Go 文件、22000 行代码、17 张数据库表、39 个业务 API、6 个子系统全部上线。这篇文章讲讲我是怎么做到的。

## 背景

公司需要一套计量计费服务（billing-svc），覆盖产品目录、订阅管理、计量引擎、费用计算、账单、支付六大子系统。架构师输出了完整的架构设计文档，包括领域模型、接口定义、数据库 ER 图、计费策略。我的任务是：把文档变成可运行的代码。

问题是——我不会 Go。

## 工具链：OpenSpec + Claude

我用了两个核心工具：

**OpenSpec** 是一套规范驱动的开发流程，核心思路是把给 AI 的输入分成两层——**How（怎么写）** 和 **What（做什么）**：

**How — 工程规范**（`docs/specs/`）。定义了 API 接口规范、响应结构规范、Go 编码风格、IAM 权限命名规范等文档。这些规范会作为上下文喂给 Claude，相当于告诉它「代码应该长什么样」。效果是 Claude 生成的代码在命名风格、响应格式、错误处理上天然统一，省去了大量 review 中反复纠正格式的成本。

**What — 变更管理**（`openspec/changes/`）。把一个大任务拆成多个带时间戳的 change（变更单元），每个 change 有独立的 `.openspec.yaml` 描述目标、范围和验收标准。开发过程就是按顺序推进 change，每完成一个归档到 `archive/`。它同时服务于 AI 和人——AI 从中获取精确的任务边界，人通过归档记录回溯完整开发时间线。

两层叠加的好处是：
- 把「写一个计费系统」这种模糊目标，拆成了「实现 catalog CRUD」「加 K8S 部署」「加单元测试」这些可执行的小任务
- 每个 change 有清晰边界，Claude 能精准理解当前要做什么
- 规范约束 + 任务边界，双重保障让 AI 生成的代码质量稳定可控

**Claude**（通过 Claude Code）是实际写代码的主力。我的工作模式是：把架构文档和当前 change 的目标喂给 Claude，让它生成代码，我做审查和调整。

## 实际开发过程

时间线大致如下：

**4月8日** — 项目初始化。Claude 根据架构文档生成了项目骨架：目录结构、Go module、Gin 路由注册、统一响应封装、错误码体系。同时完成了 6 个子系统的 spec 文档拆分，为后续开发建立清晰边界。

**4月9日** — 六大子系统的 handler → service → repository 三层代码全部生成。17 张表的 DDL、GORM model、CRUD 接口一次性产出。我的工作是对照架构文档逐个检查字段名、索引、业务约束是否对齐。同一天完成了鉴权中间件集成（RBAC + ABAC）。

**4月10日** — K8S 部署。Dockerfile、Deployment YAML、Service、ConfigMap，Claude 根据公司已有的部署模板生成，我调了镜像仓库地址和资源限制就部署成功了。同一天补了单元测试和接口调用示例文档。

**4月11-12日** — 联调测试和订阅逻辑修改。这是第一次碰到 Claude 生成的代码不能直接用的情况——订阅状态机的边界条件（过期续费、降级退款）需要结合业务理解手动调整。同时统一了公有云与私有化订阅开通流程。

**4月13-15日** — 计量引擎核心实现。配额系统改为 Redis-first 架构，实现了用量聚合、配额检查与重置。这部分逻辑较复杂，涉及分布式锁、原子预占与回滚，我先在纸上画清数据流转，再让 Claude 按流转图写代码。

**4月16日** — 实现订阅续费和在线支付。涉及到期判断、自动续费、配额重置、微信/支付宝支付适配器。效果比直接让它「实现续费功能」好得多。

**4月17日** — 工程规范检查。错误处理统一、日志格式规范（zap 结构化日志）、SQL 注入防护、并发安全审查。同时集成了 Swagger UI。

**4月19日** — 整体重构和收尾。精简文档、补充幂等测试用例、代码合并上线。

## 前端转后端，哪些地方需要自己判断

Claude 能写出语法正确、结构清晰的 Go 代码，但以下几类问题它帮不了：

1. **业务理解**。计费模式有 6 种（按量、按次、阶梯、包年包月等），每种的计算逻辑和边界条件，必须自己理解架构文档才能审查代码是否正确。
2. **数据一致性**。扣费和配额变更需要事务保证，Claude 生成的代码默认用了 GORM 事务，但跨表操作的顺序、死锁风险需要自己考虑。
3. **部署环境差异**。K8S 的 namespace、镜像仓库、ConfigMap 挂载路径，这些跟公司基础设施相关，没法指望 AI 知道。

## 前端背景有什么优势

也不全是劣势。前端经验带来了几个直接可迁移的能力：

- **分层架构的直觉**：React 的 component → hook → API 和 Go 的 handler → service → repository 本质上是同一种关注点分离
- **API 设计**：RESTful 接口的 URL 设计、状态码、分页、过滤，前端调了无数遍，自己设计时反而更知道什么好用
- **TypeScript 的类型思维**：Go 的强类型、struct、interface 对写过 TS 的人来说几乎零门槛

## 总结

这次经历让我确认了一件事：**AI 编程工具真正改变的不是写代码的速度，而是跨技术栈的门槛**。

以前前端转后端，至少要花几周学语法、学框架、踩部署的坑，然后才能开始写业务。现在这些「固定成本」被 Claude 几乎消除了——它就像一个随时在线的 Go 导师，你问它语法它教你，你给它设计文档它帮你写初版代码。

但 OpenSpec 的作用同样关键。没有它把任务拆成可控的 change，直接把整个架构文档丢给 Claude 说「帮我实现」，大概率得到的是一堆不可用的半成品。**AI 需要精确的上下文边界，OpenSpec 正好提供了这个边界。**

最终交付：6 个子系统、39 个业务 API 接口、17 张数据库表、完整的 e2e 测试（含幂等测试）、K8S 部署，从零到上线两周。代码质量经过工程规范检查，目前在生产环境稳定运行。

对于想尝试类似路径的同学，我的建议是：**别从学语法开始，从一个真实项目开始，让 AI 帮你边做边学。**
